# Project Logic

## Product Direction

The application should support two different exercise playback models:

1. **Guided exercise-along programs**
2. **Demonstration-based manual programs**

These modes solve different clinical and user-experience problems. The product should not force every exercise into one playback model.

## Core Problem

The previous seamless-loop idea used a recorded half repetition, duplicated it, reversed the second half, and joined the two parts together. This works visually because the start and end positions are identical, which makes looping smooth.

However, this approach breaks for exercises with asymmetric tempo.

Example:

- Squat down phase: 3 seconds
- Squat up phase: 1 second

If the second half is generated by reversing the first half, the eccentric and concentric phases will always have the same duration. That makes the movement clinically incorrect for eccentric exercises, tempo-based rehab, and any exercise where the down and up phases need different timing.

## Playback Modes

### Guided Exercise-Along Mode

Use this mode when the patient is expected to follow the video in real time.

This mode is useful for:

- Simple mobility exercises
- Symmetrical movements
- Exercises where exact loop smoothness matters
- Exercises where the patient should move together with the video

In this mode, seamless playback is important. The application may use loop-friendly videos or generated loops when the exercise type supports it.

### Demonstration Manual-Next Mode

Use this mode when the video is a demonstration, not a real-time follow-along timer.

This mode is useful for:

- Eccentric exercises
- Tempo-specific exercises
- Complex rehab movements
- Therapist-recorded demonstrations
- Exercises where the patient performs reps independently

The user watches the demonstration video, performs the prescribed exercise, and manually clicks **Next** when finished.

In this mode, perfect looping is less important than preserving the real movement and correct timing.

## Video Upload Logic

For demonstration-based exercises, users should upload a real exercise video instead of relying on generated mirrored reps.

Recommended upload format:

- Record 3-5 full repetitions.
- Start in the natural resting position.
- End in the same resting position if possible.
- Add a short pause before the first rep and after the last rep.
- Preserve the real tempo of the exercise.

This reduces visible loop stutter because the transition happens only after several reps instead of after every single rep.

For example, instead of uploading one 4-second squat repetition and looping it every 4 seconds, the user should upload five repetitions that last around 20 seconds total.

## Tempo Handling

Tempo should be stored and shown as exercise metadata, not inferred from generated video timing.

Examples:

- `3 seconds down, 1 second up`
- `2 second hold at the bottom`
- `Slow controlled lowering`
- `Pause at end range`

The video should demonstrate the tempo, but the application should also display the written prescription clearly so the patient understands what to do.

## Program Player Behavior

Each program exercise should define which playback mode it uses.

For guided exercises:

- The player can auto-progress based on duration, reps, or programmed timing.
- The patient follows the video while it plays.
- Loop smoothness is important.

For demonstration exercises:

- The player loops the demonstration video.
- The player displays sets, reps, tempo, rest, and therapist notes.
- The patient performs the exercise independently.
- The patient clicks **Next** manually after finishing.
- Auto-progression should not be required.

## Exercise Data Model Concept

Each exercise should be able to store:

- Exercise name
- Video URL
- Playback mode: `guided` or `demo`
- Reps
- Sets
- Rest time
- Tempo instructions
- Therapist notes
- Optional flags such as unilateral, pain guidance, or equipment

The exact database schema can evolve, but the business logic should keep playback mode explicit.

## Design Principle

Seamless looping is a visual feature. Correct movement tempo is a clinical feature.

When these conflict, the application should prioritize correct clinical instruction.

## Recommended Direction

Do not replace the exercise-along player completely.

Instead, keep both modes:

- Use **guided exercise-along** where it creates a better patient experience.
- Use **demonstration manual-next** where clinical accuracy, tempo, or therapist intent matters more than synchronized playback.

This allows the product to support both simple follow-along programs and more realistic physiotherapy prescriptions.
